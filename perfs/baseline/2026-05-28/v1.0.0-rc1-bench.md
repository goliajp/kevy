# v1.0.0-rc1 baseline bench

Permanent record of the lx64 numbers behind the
[`v1.0.0-rc1`](../../../CHANGELOG.md) tag. All measurements taken on
**lx64** (Debian trixie, 6.12.74+deb13+1-amd64, 16 cores, load avg ~2.5
with 5 background valkey containers running unrelated workloads).

Methodology mirrors the in-tree
[`bench/lx64_loopback.sh`](../../../bench/lx64_loopback.sh) /
[`bench/pubsub_loopback.sh`](../../../bench/pubsub_loopback.sh) /
[`bench/pubsub-compare/run.sh`](../../../bench/pubsub-compare/run.sh)
scripts: server pinned to cores 0-9, client/loadgen pinned to cores
10-15, host loopback (no docker bridge unless explicitly noted), each
server measured in isolation (start → bench → stop), 2 measurement
runs per backend reported.

Build: `cargo build --release` (LTO fat, opt-level 3, codegen-units 1,
panic abort, strip on).

## 1. SET / GET throughput

Workload: `redis-benchmark -t get,set -n 3000000 -c 50 -P 16 --threads 6`.

| Server | SET rps | GET rps | p50 SET | p50 GET | vs valkey-def |
|---|---:|---:|---:|---:|---:|
| **kevy io_uring (10 sh)** | **3.99 M** | **3.99 M** | 0.13 ms | 0.12 ms | **3.66× / 3.32×** |
| **kevy epoll (10 sh)** | **2.40 M** | **2.40 M** | 0.28 ms | 0.23 ms | 2.20× / 2.00× |
| valkey 9.1 default | 1.09 M | 1.20 M | 0.68 ms | 0.58 ms | 1.00× |
| valkey 9.1 io-threads=10 | 1.33 M | 1.71 M | 0.58 ms | 0.37 ms | 1.22× / 1.43× |
| redis 7.4 default | 1.33 M | 1.50 M | 0.52 ms | 0.47 ms | 1.22× / 1.25× |
| redis 7.4 io-threads=10 | 0.15 M | 0.18 M | 4.0 ms | 4.0 ms | — (docker image config-broken) |

Headline number: **kevy io_uring 4.0 M SET ≈ 4.0 M GET**, matching the
README claim. README's stated `2.67×` -c50 SET ratio is conservative —
the actual ratio in this run is 3.66×.

## 2. Pub/Sub fan-out throughput

Workload: 1 publisher → 50 subscribers, 200 k publishes, 16 B payload,
10 M deliveries total.

### 2a. Host-mode bare-metal (kevy in-tree script)

`bench/pubsub_loopback.sh`:

| Server | Delivered msg/s (best of 2) | Publish rate | Elapsed |
|---|---:|---:|---:|
| **kevy epoll (10 sh)** | **18.2 M** | 364 k/s | 0.55 s |
| **kevy io_uring (10 sh)** | 17.8 M | 356 k/s | 0.56 s |
| valkey 9.1 | 6.8 M | 137 k/s | 1.46 s |
| redis 7.4 | 8.8 M | 175 k/s | 1.14 s |

kevy ≈ **2.6× valkey 9.1** at this fan-out workload — above the
`v0.metal` baseline of 2.3×.

### 2b. Container-isolated 6-way (bench/pubsub-compare/run.sh)

Adds ZeroMQ 4.3.5 (latest stable), Zenoh 1.9 (latest), Aeron 1.45 (via
rusteron-client 0.1.163; embedded media driver in IPC mode).

| System | Transport | Best msg/s | vs valkey-def |
|---|---|---:|---:|
| **Aeron 1.45 IPC** | shared memory | **20.2 M** | 3.01× |
| **kevy (10 sh)** | TCP loopback (RESP) | **14.2 M** | 2.12× |
| ZeroMQ 4.3.5 | tcp:// loopback | 9.1 M | 1.36× |
| redis 7.4 | TCP loopback (RESP) | 8.6 M | 1.28× |
| valkey 9.1 | TCP loopback (RESP) | 6.7 M | 1.00× |
| Zenoh 1.9 | peer mesh | 3.3 M | 0.49× |

**Notes:**

- **Aeron IPC is the absolute ceiling** — shared-memory log buffers, no
  kernel network stack, no protocol overhead. Any TCP-based system is
  structurally below this number.
- **kevy beats ZeroMQ ~1.6×** on the same TCP loopback transport despite
  speaking the full RESP protocol. The win comes from kevy's per-shard
  fan-out amortising the dispatch cost (ZMQ's client-side PUB has to
  serialize to every SUB socket individually).
- **kevy in-docker (14.2 M) is ~22 % below bare-metal (18.2 M)**.
  Container cgroup scheduling + bridge overhead applies to every backend
  equally; the relative ratios hold.
- **Zenoh underperforms** here because the 50-way fan-out hits its
  tokio channel path; Zenoh is optimised for query/put + sparse
  subscribers, not broadcast.

## 3. Memory footprint

Workload: 1 M random-keyspace SETs of 16-byte values (collides to ~632 k
unique keys due to birthday-paradox in `redis-benchmark -r 1000000`).
RSS read from `/proc/PID/status:VmRSS`. Same dbsize across all systems
for apples-to-apples per-key comparison.

| Server | empty RSS | full RSS | RSS delta | **per-key delta** | INFO used_memory |
|---|---:|---:|---:|---:|---:|
| **kevy 1 shard** | **2.5 MB** | 78 MB | 76 MB | **120 B** | 57.9 MB |
| kevy 10 shards | 3.6 MB | 106 MB | 103 MB | 163 B | 5.8 MB (one shard) |
| valkey 9.1 | 17.9 MB | 56 MB | 39 MB | **64 B** | 38.8 MB |
| redis 7.4 | 15.9 MB | 83 MB | 70 MB | 113 B | 62.1 MB |

**v1.0.0-rc1-time win:** `Entry` shrunk 64 → 48 bytes (commit `e32c5c5`)
dropped kevy 1sh from **146 → 120 B/key (-18 %)**, basically matching
redis 7.4 at 113 B/key. Still 1.9× heavier than valkey 9.1 (was 2.3×);
the remaining gap is fundamental — kevy's `SmallBytes` inline SSO and
7/8-load KevyMap deliberately trade per-key memory for CPU cycles
(eliminates the 2nd cache miss on GET / overlapping hashbrown bucket
layout). For 16 B values where overhead is dominant this is a 1.9×
tax; for typical Redis use (longer values) it amortises away.

**Empty RSS** is the unsung win: kevy at 2.5 MB vs valkey 18 MB / redis
15 MB. 7× smaller idle footprint — matters a lot for embedded /
container density scenarios.

## Conclusion

All three classic perf indicators lead vs valkey 9.1:
- **CPU**: kevy 3.7× on -c50 SET, 3.3× on -c50 GET (io_uring)
- **Pub/sub**: kevy 2.6× delivered msg/s
- **Empty memory**: kevy 7× smaller idle RSS

Per-key memory is the only metric where kevy is heavier (1.9×) — the
design-level trade-off for the CPU wins. Documented in the v1.x
stability commitment so users opting for kevy know the shape.

Ready for the RC period.
