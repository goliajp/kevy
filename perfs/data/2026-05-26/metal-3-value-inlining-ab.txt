== bench/metal_keyspace.sh — GET -c50 -P256, random over keyspace ==

Same script, same env, populate via `-r N -n N`, then GET `-r N -n 10M`.
Default 3-byte payload (well within the 22-byte inline SmallBytes capacity,
so SET stores the value inline in the bucket — exactly the path metal-3
targets).

  keyspace        baseline(metal-1)   metal-3 (this) Δ vs baseline
  1                4,756,781 rps      4,619,624 rps   -2.9%
  100,000          3,387,832 rps      3,554,455 rps   +4.9%
  1,000,000        2,713,366 rps      2,926,637 rps   +7.9%
  10,000,000       2,283,757 rps      2,577,702 rps   +12.9%

→ The memory wall flattens. The cache-hot single-key path takes a small
  hit (extra branch in as_slice/is_inline on top of the dispatch), but the
  real-workload tail (10M keys, DRAM-bound) gains +12.9% — exactly the
  prediction: SmallBytes kills the value pointer-chase 2nd cache miss the
  bucket-probe used to require for small string values.

→ metal-2 only measured the 10M-key column. vs that anchor: 2,008,073 →
  2,577,702 = **+28.4%** on the headline real-workload number. (metal-2's
  bench preload used `-r 10M -n 20M` which yields ~8.6M DBSIZE; metal-1
  baseline column uses metal_keyspace.sh's `-r N -n N` form which yields
  ~6.32M DBSIZE at N=10M. We trust the like-for-like baseline column above
  for the headline; the metal-2→metal-3 number is informational.)


== RSS @ ~8.6M keys (preload `-r 10M -n 20M` — matches box-coll's protocol) ==

                          DBSIZE        VmRSS
  metal-1 baseline        n/a           n/a    (metal-baseline only sampled at 864k)
  metal-2 (box-coll)      ~8.6M         1,608,496 kB  (1.61 GB)
  metal-3 (this)          8,646,833     1,469,428 kB  (1.47 GB)
  Δ vs metal-2                                       **-138 MB / -8.6%**

→ Small string values (3 B) now ride inside the bucket Entry instead of
  inheriting Vec's 24 B + heap allocation. Per-key savings ≈ 16 bytes ×
  8.6M ≈ 137 MB — matches the measurement.


== Binary size ==

  metal-1 baseline (metal-baseline.txt):  ~0.4 MB stripped (mac build?)
  metal-3 on lx64 (release stripped):     670,672 bytes   (655 KB)

→ The baseline figure may be from a different build environment; on lx64
  the stripped ELF is 655 KB. Increment vs the "0.4 MB" baseline (+255 KB)
  is larger than the kevy-bytes module alone would explain; likely the
  baseline figure was measured on mac (different code-gen + smaller libstd
  link). Not treating this as a metal-3 axis regression — we'll
  re-establish a clean lx64 binary-size baseline in a future checkpoint.
  Stripped lx64 binary verified via `file` ("stripped") and `nm` ("no
  symbols").

== Judgement: KEEP ==

  perf axis (real workload): +12.9% @ 10M-key GET (and monotone
    improvement as keyspace grows past L2/L3).
  perf axis (cache-hot 1-key): -2.9% — accepted; not representative
    of real workload; the cost is one extra branch on the hot path.
  mem axis: -8.6% RSS (-138 MB at 8.6M keys), consistent with
    per-Entry savings of ~16 bytes for small-string values.
  size axis: deferred (cross-env baseline; will re-anchor on lx64).

→ Memory-wall lever #1 lands as designed. Two more sub-stones remain in
  this category: v0.metal-4 (self-built kevy-map → exposes bucket
  addresses) and v0.metal-5 (software prefetch + cache-conscious bucket
  layout, the direct hammer on the bucket-probe DRAM miss).
