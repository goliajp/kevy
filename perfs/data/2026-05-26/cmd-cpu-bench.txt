Command-path CPU component bench + system A/B — the "polish the stones" round.
crates/kevy/examples/bench_cmd.rs (mac, indicative); system A/B on lx64 io_uring
single-shard (core 0) + 15-core client, GET, host-loopback.

COMPONENT (mac, parse + dispatch, ns/op):
  before encode_bulk reserve:           after reserve:
    dispatch GET (hit)   100 ns           dispatch GET (hit)    39 ns   (-61%)
    dispatch SET          44 ns           dispatch SET          45 ns
    dispatch INCR         91 ns           dispatch INCR         91 ns

  => GET reply was realloc-bound (fresh empty Vec grown 0→8→16→32 by encode_bulk);

SYSTEM A/B (lx64, single-shard io_uring, develop vs feature):
  -P16  GET: develop 2.21/2.26M  feature 2.21/2.11M   neutral
  -P64  GET: develop 3.40/3.41M  feature 3.44/3.41M   neutral
  -P256 GET: develop 3.68/3.69M  feature 3.74/3.48M   neutral

KEY FINDING: the component -61% does NOT move system throughput at ANY pipeline
depth. The single-shard server is REACTOR/IO-bound, not command-CPU-bound: of
~265 ns/cmd at the single-shard ceiling, command CPU is only ~76 ns (now ~half on
GET); the other ~190 ns is reactor/io_uring (completion handling, recv copy,
seq-ring fold/PendingSlot/drain, write arming). The tight-loop micro-bench
amplified the realloc cost (cold allocator); in the server the allocator is warm
and other work amortizes it.

=> encode_bulk reserve KEPT (free, correct, component-proven, never hurts), but
   command-path CPU polishing has hit diminishing system returns. To raise
   per-core throughput, the REACTOR path is the stone (hard to dissect without a
   profiler — perf is not installed on the shared lx64). The per-core ceiling
   (3.77M = ~1.5x valkey's total) and the multi-core scaling headroom (needs a
   2nd load-gen box) stand.
