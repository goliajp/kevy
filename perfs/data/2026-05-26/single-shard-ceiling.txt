Single-shard per-core ceiling — lx64, 1 shard on core 0, client cores 1-15
(15 cores, so the load generator is definitively NOT the bottleneck), GET
-c50 -P256, host-loopback. redis-benchmark 8.0.2, steady-state overall rps.

  [1sh-epoll]    GET overall = 3,766,019 rps
  [1sh-io_uring] GET overall = 3,758,742 rps

Reading:
- kevy's per-core GET ceiling is ~3.77M (≈265 ns/command). That is already ~1.5x
  valkey 9.1's TOTAL throughput (valkey command execution is single-threaded,
  ~2-2.5M).
- epoll ≈ io_uring at deep pipeline (both saturate the core); io_uring's edge is
  at lower -P / higher connection counts.
- The earlier 4-shard 7.2M (=1.8M/shard) was CLIENT-bound (12-core client), not a
  server limit: one shard alone does 3.77M. So the dominant untapped headroom is
  MULTI-CORE SCALING — ~10-13 usable shards × 3.77M = 38-49M GET/s on this box —
  which no single-box co-located client can drive. Measuring/exploiting it needs a
  dedicated (ideally multiple) load-gen machine.
- Per-core, ~265 ns/cmd: ~110 ns command CPU (parse 40 + store 28 + encode 5 +
  route 10 + fold/drain ~30) + ~150 ns reactor/io_uring. The per-core reactor cost
  is the only single-box-measurable lever left, and its remaining knobs
  (registered files/buffers, per-iteration scan trimming) are complex + modest.
