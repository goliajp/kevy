→ Only the 100k column shows a clean improvement (+3.8%); 1-key and 10M
  are flat within noise; 1M shows a small dip but the run spread is
  ~250k/3.05M = 8.2% (above the cross-stone delta), so this is noise.

→ Cumulative vs metal-1 baseline holds: 100k +18.7%, 1M +11.0%,
  10M +24.5%. The 1-key cache-hot column is still down ~7% from baseline
  — see the metal-6 note; further metal stones (esp. metal-8 dispatch
  redo) will re-test whether this is fixable.

→ Why so modest: `find_crlf` is called only for the short `*N\r\n` header
  and each `$len\r\n` length-line — typically 3-5 bytes each. The 8-byte
  SWAR loop body is rarely entered; most scans run only the scalar tail.
  The trailing bulk-data `\r\n` is checked by direct slice compare, not
  `find_crlf`, so the long-data path doesn't see SWAR either.

== Judgement: KEEP ==

  perf axis: +3.8% on 100k column (small clean win); 1-key, 1M, 10M
    within noise band. No axis ≥5% paired regression.
  mem axis : neutral (no allocation change).
  size axis: neutral.

→ Lesson for the plan: parse_command's CPU is not concentrated in
  `find_crlf` after metal-6; it's spread across `push`/`extend_from_slice`
  (the actual byte copy into scratch.buf), length-line bytes, etc.
  Further parse-side gains need either avoiding the byte copy entirely
  (true borrowed argv with lifetime tied to conn.input — a bigger
  refactor than metal-6 did) or moving to RESP3 binary length fields.
  Both are out of metal-7's scope. Phase A's biggest remaining lever
  is now squarely `start_command` (still 18% of CPU) — promote
  v0.metal-8 dispatch redo next.
