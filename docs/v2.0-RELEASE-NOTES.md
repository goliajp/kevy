# kevy v2.0 — release notes (DRAFT)

**Status**: Draft prepared at v1.54 ship (2026-06-30). Updated through
the RC cycle (v1.55 → v1.59) as ecosystem-battle feedback comes in.
This file becomes the canonical v2.0 release notes on the final
`v2.0` tag.

---

## TL;DR

**v2.0 makes kevy industrial-grade.** It is not "kevy with one big
feature" — it is the result of a sustained chaos / soak / battle-test
arc that hardened every failure-mode, observability path, cluster
correctness invariant, and ecosystem client integration that production
deployments depend on.

If you are running kevy v1.x today: upgrading to v2.0 is **drop-in**.
No config migration, no on-disk format change, no API break. Every
behaviour change is a strict superset of v1.x semantics.

If you are evaluating kevy: v2.0 is the version we recommend for first
production deployment. Earlier v1.x ships were correct; v2.0 is the
first version we have empirically proven survives the entire failure-
mode surface area.

## What v2.0 changes vs. v1.45 (the last pre-v2-arc ship)

### Phase A — failure-mode robustness

- **RESP parser** validated against 1 M random byte streams via the new
  std-only `kevy_resp::fuzz` harness (v1.36). 0 panics, 0 crashes,
  every malformed input produces a well-formed `-ERR` reply. The error
  catalogue is in [`docs/error-replies.md`](error-replies.md).
- **Connection cap** enforced via `[server].max_clients = N` (v1.37).
  Per-shard limit = `ceil(N / shard_count)`. Excess conns rejected,
  `rejected_connections` counter increments. Verified empirically at
  50-conn cap: 50 accepted, 51st rejected, no panic.
- **Resource exhaustion** handled gracefully (v1.38). FD-exhaust honors
  RLIMIT_NOFILE without crashing. Disk-full → SIGXFSZ kernel-kill +
  clean restart from AOF (the restart-recovery contract is what is
  guaranteed; in-process write survival is not).

### Phase B — operability + observability

- **SIGTERM graceful drain** (v1.39). Async-signal-safe handler installed
  via `kevy_sys::install_signal_handler(SIGTERM, ...)`. Empirical:
  192 k pre-stop ACKs / **0 lost** / 0.08 s drain time at sustained
  load.
- **Backup / restore CLI** (v1.40). `kevy-cli backup` packs the entire
  data dir into a single std-only KEVYBKP1 archive; `kevy-cli restore`
  unpacks. Race-safe: pack zero-pads if a file shrinks mid-read.
- **Prometheus `/metrics` endpoint** (v1.41). Pure-std HTTP/1.1, opt-in
  via `[metrics] listen_port`. Standard `text/plain; version=0.0.4`
  exposition format.
- **Audit log** (v1.42). Optional `[audit] log_path = "..."` appends a
  tab-separated record per `CONFIG SET` / `CONFIG REWRITE` / `DEBUG`
  command. 256 B record cap; atomic `O_APPEND` writes.

### Phase C — cluster correctness under chaos

- **Cluster-topology** (v1.43), **multi-node peer formation** (v1.44),
  and **kevy-scope MISDIRECTED** (v1.45) chaos tests all SIGKILL the
  primary node and assert the survivor stays alive + answers PING.
- **Network partition** (v1.46) — burst-abandon partial RESP frames,
  half-close patterns, 1000-conn reconnect storm. Empirical:
  **1000 / 1000 storm conns in 0.10 s = 10 k conn/s, zero refusals**.
- **AOF compat matrix** (v1.47) — hand-crafted v1.0-vintage RESP AOF
  (50 SET + 10 INCR + 10 LPUSH + 10 HSET + 10 SADD + 10 ZADD + torn
  trailer) replays cleanly, torn trailer discarded silently. Any
  `v1.x → v2.0` rolling upgrade is data-safe.

### Phase D — large-scale E2E

- **Multi-tenant E2E** (v1.48) — 5 tenants × 4 writers × 250 SETs
  = **5000 ACKs in 0.05 s, fairness skew = 0, zero cross-tenant leak**.
- **Burst / ramp / realistic workload** (v1.49) — 4-phase traffic shape
  with realistic op-mix. **Burst phase = 10 004 ACKs in 1 s = 10 k
  ops/s sustained**, post-burst memory 5.9 MiB (cap 8 MiB).
- **Long-running soak** (v1.50) — opt-in via `KEVY_SOAK_SECS=86400`.
  Memory-slope OLS leak detector. 30 s smoke: **143 k ACK/s, slope
  4.7 KiB / sample = 56× under the leak cap**.
- **v2 acceptance baseline doc** (v1.51) at
  [`docs/v2-acceptance-baseline.md`](v2-acceptance-baseline.md) —
  catalogs every test + empirical baseline.

### Phase E — ecosystem battle-test

- **Jedis 5.x** + **StackExchange.Redis 2.x** (v1.52) — both golden
  paths pass. Jedis pipeline of 100 mixed commands replied with
  exactly 100 well-formed RESP frames; StackExchange.Redis HELLO 3
  + CLIENT NO-EVICT + 5-key MGET all clean.
- **go-redis v9** + **redis-py 5.x** (v1.53) — both golden paths
  pass. go-redis CLIENT INFO returns the full 141-byte bulk (16
  fields); redis-py WATCH / MULTI / INCR / INCR / EXEC commits
  atomically; pub/sub cross-conn round-trip delivers `message` frame
  to the subscriber.

## The chaos suite

`cargo test -p kevy --release -- --ignored --nocapture` runs the full
suite in under 60 s wall-clock on a Mac M2 Pro. The 24 h soak gate is
opt-in:

```text
KEVY_SOAK_SECS=86400 cargo test -p kevy --test soak_long_running_chaos \
    --release -- --ignored --nocapture
```

Every test in the suite is shipped, gated `#[ignore]`, runs against
the kevy release binary, and has a strict empirical baseline
documented in [`docs/v2-acceptance-baseline.md`](v2-acceptance-baseline.md).

## What v2.0 explicitly does NOT change

- **Performance.** v2.0 is throughput-identical to v1.45 across every
  measured axis. The v2 arc was hardening, not micro-optimization.
  (Perf headlines are in [`bench/REPORT.md`](../bench/REPORT.md).)
- **API surface.** Every v1.x command works identically in v2.0.
  No deprecations, no soft-removed flags.
- **Wire format.** AOF / RDB / replication wire formats are unchanged
  from v1.x. Rolling upgrade is data-safe (validated by the v1.47
  AOF compat chaos test).
- **0-dependency principle.** v2.0 ships with the same three carved
  exemptions as v1.x (kevy-client-async / kevy-lua / kevy-lua-host);
  the default server stack remains 0 third-party deps.
- **AUTH / TLS.** Permanently out of scope per project charter. v2.0
  is single-DC, intranet-only; production deployments terminate TLS
  at an upstream proxy (envoy, nginx, AWS NLB) the same way they
  would for v1.x.

## Known open findings (non-blocking)

These observations surfaced during the v2 arc and are filed as
v1.x.x patch candidates. None block v2.0 ship:

1. v1.43.x: `MGET` cross-slot returns multi-bulk nils instead of
   `-CROSSSLOT` error.
2. v1.44.x: `kevy-elect cluster_known_nodes` reports 0 even after
   peer formation.
3. v1.45.x: `-MISDIRECTED` reply quotes the elect_port instead of
   the main client port.
4. v1.33.x: Linux replication chaos test fails to fire (needs
   Linux-side repro).
5. v1.34.x: 1 h opt-in soak run on lx64 not yet executed.
6. v1.49.x: `INFO memory` reports `used_memory:` empty when the
   keyspace is empty (semantics observation, not a leak).
7. v1.38.x: `SIGXFSZ` handler not installed — restart-from-AOF is
   the contract, not write survival.
8. v1.52.x: `CLIENT SETNAME` is a documented stub; `CLIENT GETNAME`
   always returns empty bulk. Jedis records the name client-side
   so app correctness is unaffected; `CLIENT LIST` observability
   sees `name=""`.

Each open finding is referenced in the CHANGELOG entry of the
release where it was discovered.

## Upgrade procedure

Drop-in. From v1.x:

```bash
# 1. Stop the v1.x kevy process. SIGTERM triggers the v1.39 graceful
#    drain — wait for it to exit (~0.08 s under load).
kill -TERM <kevy-pid>

# 2. Replace the binary with v2.0.
cargo install kevy --version 2.0.0
# or via package manager / Docker image (see deployments docs)

# 3. Start the v2.0 kevy process with the same config file and data
#    dir. AOF replays cleanly (the v1.47 compat test proves this).
kevy --config /etc/kevy/kevy.toml

# 4. Verify:
redis-cli -p 6379 PING
redis-cli -p 6379 INFO server | grep redis_version
```

If you run replication: upgrade the replicas first, then the primary
(standard rolling-upgrade order).
