## 🔴 Findings — critical DoS bug found by fuzz

During the parse_reply fuzz smoke (60 seconds), libfuzzer surfaced
crash-`4c4ee6...`:

**Input** (38 bytes): `*000000000000000086433320554209243...`

**Root cause**: `parse_array_reply` does
`Vec::with_capacity(count as usize)` where `count` is from
`parse_int(&buf[1..hdr_end])` returning i64. An attacker-controlled
header `*9999999999999\r\n` produces count ≈ 8e18, fits in i64, passes
the `< 0` guard, and `Vec::with_capacity(8e18_usize)` panics with
capacity overflow. **Any RESP server that uses kevy_resp::parse_reply
on untrusted input is panic-DoS-able by a single 14-byte request.**

(kevy itself uses `parse_command`, not `parse_reply`; `parse_reply` is
only on the client side (kevy-resp-client / kevy-cli) — so the
DoS surface is "RESP client parses adversarial server response" not the
server. Still a real bug if a malicious RESP server hits a kevy-cli /
kevy-resp-client user.)

**Fix (committed in this audit)**:
```rust
let cap = (count as usize).min(buf.len().saturating_sub(pos));
let mut items = Vec::with_capacity(cap);
```
Each Reply item costs ≥ 1 byte (a CRLF), so a real array of N items
requires ≥ N bytes in the buffer. Cap initial capacity by remaining
buffer size; let `push` amortize-grow if the genuine count is higher AND
buffer keeps producing items.

**Regression coverage**: the crash artifact is retained at
`fuzz/artifacts/parse_reply/crash-4c4ee6...` and replayed every fuzz
run. Re-fuzzed for 60s post-fix: 0 crashes (4.65M executions).
