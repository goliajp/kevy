# kevy v1.0 boundary — "self-host prod-ready" 完整计划

Locked: 2026-05-27, by user.

## 目标

让 kevy 在 **dev / docker-compose 内部 / embedded / cache** 4 个场景生产
可用。不追求公网部署 / cluster / HA — 这些走 v2+ 或 cloud-native
sidecar (k8s StatefulSet, sentinel-style operator) 路径。

## 已锁定的非目标（见 [`.claude/scope-decisions.md`](.claude/scope-decisions.md)）

- **AUTH / TLS** — v0.3+；目标场景 trust boundary 在网络层（docker
  network / k8s pod network / embedded in-process）
- **Replication** — 推迟；目标场景里 crash-safe persistence 已够，
  HA 走 k8s StatefulSet
- **Cluster mode** — 永久 OUT，CLAUDE.md 已钉
- **Bare-metal MCU (`no_std`)** — 永久 OUT，"IoT" 限定 Linux SBC

## 已经在 v0.1.x 落地的（v1.0 不重做）

- 5 数据类型 (String/Hash/List/Set/ZSet) + pub/sub + transactions +
  cross-shard multi-key
- RESP2 protocol 全 + 94-cmd valkey parity (compat3.sh 验证)
- kevy-uring at kernel floor (148 ns vs liburing 152 ns)
- kevy-map at noise-floor parity vs hashbrown (P20 branchless set_meta)
- kevy-hash at top tier (P21 two-stream pipelined)
- kevy-resp 2.6× faster than hiredis
- e2e lx64 metal: kevy-uring 3.99M SET / 3.99M GET rps, 2-3× valkey-iot
- bind 默认 loopback + 非 loopback 启动 WARN (commit `f5477c4`)
- Snapshot v2 + AOF (基础持久化)

### Wave 2: 防 OOM + 防数据丢 — 2.5-3 周

**任务 #1: Maxmemory + 7 eviction policies** (~1.5 周)

- 实现 7 个 policy：`noeviction` / `allkeys-lru` / `allkeys-lfu` /
  `allkeys-random` / `volatile-lru` / `volatile-lfu` /
  `volatile-random` / `volatile-ttl`
- LRU/LFU 用 Redis 的 24-bit clock approximation（不要完整 LRU 链表）
  + sample-based selection（每次 evict 采样 N 个 key 选最旧）
- `used_memory` 维护：thread-local 累加（thread-per-core 模型免 atomic）
- Hot-path: writes 上加 `if used_memory > maxmemory_target` 检查
- **性能 trade-off**: 默认 `maxmemory = 0` (unlimited) → check 走 dead
  code elim，0 写路径开销；显式 set 后 -1-3% 写吞吐
- 暴露 `MEMORY USAGE <key>` / `MEMORY STATS`
- Bench: 加 maxmemory 工作负载到 perf_gate 测试

**任务 #2: TTL background reaper** (~3 天)

- 按 Redis `activeExpireCycle` 标准：每 100ms 取 20 个有 TTL 的 key
  采样；过期 → 删；批内过期率 > 25% → 继续扫；总 CPU ≤ 25% time
- thread-per-core 模型：每 shard 跑自己的 reaper（小 task 走 reactor
  timer）
- embedded 模式：spawn background thread (default) OR caller-driven
  `Store::tick()` API (cfg toggle，必需给 WASM 用)
- Lazy expiry 那条路径保留 — reaper 是补充不是替换

**任务 #3: AOF rewrite (BGREWRITEAOF) + appendfsync** (~1 周)

- `appendfsync` 三档 (`always` / `everysec` / `no`)，默认 `everysec`
- `everysec` 模式：后台 fsync thread 每秒一次（不阻塞 hot path）
- `BGREWRITEAOF`: 把当前 in-memory state dump 成一个新的、紧凑的 AOF
  (`SET key value` 序列代替历史增量)，rename atomically
- 自动触发：当前 AOF size ≥ `auto_aof_rewrite_percentage` × last-
  rewrite size && ≥ `auto_aof_rewrite_min_size`（默认 100% + 64mb）
- 写并发安全：thread-per-core 模型下，rewrite 期间老 AOF 继续 append；
  rewrite 完成后 swap

**任务 #4: 持久化 crash-safe verify** (~3 天)

- 启动期 audit：AOF + snapshot 加载 ordering 正确（snapshot first，then
  replay AOF tail since snapshot）
- Power-loss simulation：kill -9 mid-write，验证下次启动数据完整 +
  无 corruption
- 加 `bench/crash-test.sh`：循环 SET / kill / 启动 / verify N 轮
- 文档化：默认配置下 `everysec` 最多丢 1 秒数据（Redis 也一样）；
  `always` 模式 0 数据丢失但 -50% 吞吐

**Wave 2 完工 trigger**: 4 个任务全过 + crash-test 跑 100 轮 0 失败 +
perf_gate 验证 maxmemory off 时 0 退化、on 时 ≤ -3%。

## Config 设计（kevy-config crate）

### 文件位置 + precedence

Top wins:
1. CLI flag (`--config PATH` 或 `--bind X`)
2. Env var (`KEVY_CONFIG=PATH` 或 `KEVY_BIND=X`)
3. Auto-detect 文件（按序首次命中）：
   - `./kevy.toml`
   - `$KEVY_DIR/kevy.toml`
   - `/etc/kevy/kevy.toml`
4. Built-in defaults

### TOML schema

```toml
# kevy.toml — kevy server config
# All values optional; built-in defaults apply if missing.

[server]
bind         = "127.0.0.1"   # IPv4; "0.0.0.0" = all interfaces (triggers WARN)
port         = 6004
threads      = 0             # 0 = auto (CPU count)
data_dir     = "."           # snapshot + AOF location

[persistence]
aof                          = true       # append-only file
appendfsync                  = "everysec" # "always" | "everysec" | "no"
auto_aof_rewrite_percentage  = 100        # BGREWRITEAOF when ≥ 2× last
auto_aof_rewrite_min_size    = "64mb"     # never auto-rewrite below this

[memory]
maxmemory         = 0                 # 0 = unlimited; else "256mb" / "2gb"
maxmemory_policy  = "noeviction"      # one of 8 policies

[expiry]
hz       = 10                # TTL reaper Hz (10 = every 100ms)
sample   = 20                # keys sampled per cycle

[log]
level    = "info"            # "trace" | "debug" | "info" | "warn" | "error"
output   = "stderr"          # "stderr" | "stdout" | <file path>
```

### Hot-settable matrix

| Setting | `CONFIG SET` |
|---|:---:|
| server.bind / port / threads / data_dir | ❌ restart |
| persistence.aof | ❌ restart |
| persistence.appendfsync, auto_aof_rewrite_* | ✅ |
| memory.maxmemory, maxmemory_policy | ✅ |
| expiry.hz, sample | ✅ |
| log.level | ✅ |
| log.output (stdout/stderr only) | ✅ |
| log.output (file path) | ❌ restart |

`CONFIG SET` 不可改的字段 → `ERR can't be changed at runtime,
restart required` (匹配 Redis 行为)。

### CONFIG REWRITE 语义 (v1.0)

- 写回**最初加载的那个**文件路径
- 没用文件启动（CLI/env/default only）→ `ERR no config file in use`
- v1.0 重写成标准模板格式 → **打印 WARN "your inline comments will
  be lost"**
- v1.x 升级：round-trip preservation (保留注释 + 字段顺序)
- 写法：临时文件 + rename，atomic

### 失败处理

- TOML 解析错误 → 启动失败，stderr 报 `line N col M: <error>`
- 不允许 silent default-fallback
