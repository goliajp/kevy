# kevy / valkey / redis matrix bench — 2026-06-21

4-way head-to-head: kevy-server / kevy-embedded / valkey 9.1.0 / redis 8.8.0.
All servers built FROM SOURCE on the same lx64 host with the same gcc 14.2.0,
default jemalloc 5.3.0, no TLS, no persistence. Loopback. Server pinned
cores 0-9; client core 10 (multi-conn: cores 10-13).

- **kevys** = kevy v1.24.1-pending (server mode, io_uring reactor, 10 shards)
- **kevye** = kevy v1.24.1-pending (embedded mode, in-process, single-threaded)
- **valkey** = 9.1.0 from source (`--io-threads 10 --io-threads-do-reads yes`)
- **redis** = 8.8.0 from source (`--io-threads 10 --io-threads-do-reads yes`)

Host: lx64 (Intel i7-10700K Comet Lake, 16 cores, Linux 6.12,
mitigations=off). Single-run numbers (band ±2-5 % from box noise).

## kevy-embedded (in-process, no network, no protocol)

| op          | rps           | ns/op |
|-------------|---------------|-------|
| SET (overwrite) | **7 229 878** | 138   |
| GET (hit)       | **9 125 072** | 110   |
| GET (miss)      | **43 392 459**|  23   |
| INCR            | **6 131 126** | 163   |
| DEL             | **6 099 742** | 164   |

Single-thread, direct method calls. Sets the userspace data-path ceiling
without network/protocol cost. 70-90× the TCP numbers — that gap is
the wire layer + reactor + parsing + reply formatting + TCP send/recv +
kernel netstack.

## Key takeaways

1. **kevy wins where latency matters most.** c1-P1 (single connection,
   no pipelining) is the round-trip-bound case that matches the typical
   serial-client / cache-hit pattern. kevy SET is +46 % vs valkey and
   +128 % vs redis. The post-v1.24-chain perf work targeted exactly
   this case.

2. **valkey + kevy are essentially tied at concurrency.** c50-P1 / c100-P1
   / c50-INCR / c50-MGET — all within 6 % of each other. The lever shifts
   from per-op overhead to per-message protocol+kernel cost shared by
   both implementations.

3. **valkey edges out on c50-P16 GET (2.67 M vs 2.40 M).** Pipelined GET
   reply formatting is the gap. → see `LESSONS-FROM-VALKEY.md`.

4. **redis 8.8 trails both meaningfully.** Despite shipping `io-threads`
   in 8.x, the impl appears to have less head-room than valkey's. At
   c1-P1 it's ~half of kevy's throughput. Most likely candidates: less
   aggressive busy-poll, mutex placement, or single-threaded command
   dispatch even with io-threads on.

5. **kevy-embedded sets the ceiling at ~9 M GET ops/s in-process.**
   That's the gap network costs us: ~9.1 M → 83-101 k on TCP loopback
   = ~99 % overhead in network+protocol+kernel for the c1 case. Most
   of the remaining lever is on the kernel side, not userspace.

## Why redis-benchmark for all four (vs Rust client)

For a fair comparison the SAME client hits all four servers. redis-benchmark
8.8.0 (built alongside redis) is the canonical Redis-protocol client; it's
what valkey + redis would use for their own headline numbers. The kevy
Rust client (`kevy-client`) outperforms redis-benchmark in the c1-P1 case
(adds another ~10-20 % when both run kevy), but using it would mean
benching valkey/redis with a different client than they were measured by
their own teams — apples vs oranges. We stick with redis-benchmark for
the headline matrix.

## Reproduce

```bash
# On lx64 with redis + valkey source built under /root/srcbench/:
bash /root/kevy/bench/matrix.sh full
```

Source build:

```bash
cd /root/srcbench
git clone --depth 1 --branch 8.8.0 https://github.com/redis/redis.git
git clone --depth 1 --branch 9.1.0 https://github.com/valkey-io/valkey.git
(cd redis && make -j16 BUILD_TLS=no USE_SYSTEMD=no)
(cd valkey && make -j16 BUILD_TLS=no USE_SYSTEMD=no)
```
