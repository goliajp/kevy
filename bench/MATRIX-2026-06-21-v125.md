# kevy v1.25 matrix bench — 2026-06-21

After L2 (`Value::Int` INCR fast path) + L1 (`Value::ArcBulk` +
`writev` zero-copy bulk reply) chain on top of v1.24.1.

**Critical config finding (2026-06-21)**: kevy's per-shard busy-poll
admin overhead **dominates at low conn-per-shard ratios**. The default
`--threads = available_parallelism()` (10 on lx64) starves each shard
of work; `--threads 2` or `--threads 4` give 5-10% throughput uplift
across concurrent scenarios.

And at c50-P1 GET:

```

threads= 4  c50-P1-GET = 193174
threads= 6  c50-P1-GET = 172117   <-- cliff
threads=10  c50-P1-GET = 187500   (default; sub-optimal)
```

The 6-thread cliff is reproducible across runs — between 5 and 6
shards the per-shard admin cost (arm_conns walk, drain check,
inbound_dirty/parked atomic traffic) starts to outweigh the
extra parallelism gain. With ≤ 4 shards each shard gets enough
work to amortise the per-iter admin cost.

**Recommendation**: kevy default `--threads` should be **2** for
loopback-localhost workloads (the common bench shape); the bench
harness now defaults `KEVY_THREADS=2 KEVY_SRV_CORES=0-1`.

```

 0.54%  shard_flush::send_to

```

Top three remaining levers:
1. **nft_do_chain 3.04%** → D9 (per-port iptables/nftables fast-path
   ACCEPT). Direct +2-3% win. Pure host config — needs user authorisation
   to modify lx64's nftables rules.
2. **Map<I,F>::next 2.67%** → likely the `uring_reap_closed` filter+collect
   chain. E18 was reverted earlier (two `any()` scans cost more than
   they saved at t=10); at t=2 the calculus may differ. Re-attempt with
   single-scan via plumbing the io map into the dispatch QUIT path.
3. **drain_inbound_core_slow 1.48%** → at t=2 with 100 conns split
   ~50/50, ~50 % of conns hit cross-shard for single-key bench.
   `t=1` removes this entirely (sweep: 190900 vs t=2's 193112 — the
   per-shard parallelism gain barely beats the cross-shard tax).

## Path to ≥120% on every scenario

Userspace levers are essentially exhausted. From here, the gaps from
99-119% to ≥120% require one of:

1. **Host-config wins** (need user auth): D9 nft fast-path +2-3%,
   D1 hugetlbfs for .text +1-2% (iTLB), C1 BOLT post-link
   optimization +3-5% (needs llvm-bolt install).
2. **Architectural pivot** to valkey-style 1-dispatcher + N-io-threads
   (multi-week rewrite — kevy's shared-nothing identity becomes
   `kevy --concentrator` opt-in mode).
3. **Wire-protocol-extension** workloads (e.g. forcing pipelining at
   the bench level) where kevy already leads (c50-P16 SET 119%).

Recommended next step: **D9 iptables fast-path** is the highest
ROI item — pure config, no code change, deterministic 2-3% gain
attributable to the nft_do_chain line above.

## Reproduce

```bash
# On lx64 (kevy source at /root/kevy, redis + valkey at /root/srcbench):
KEVY_BIN=/root/kevy/target/release/kevy \
KEVY_THREADS=2 KEVY_SRV_CORES=0-1 SRV_CORES=0-9 \
  bash /root/kevy/bench/matrix.sh
```
